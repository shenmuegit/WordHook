# HN 英语学习浏览器扩展 · 实施方案（含 Anki 导出）

> 定位：在 Hacker News 上**划词 → LLM 分析(单词/语法/句式) → 悬浮窗显示 → 语音朗读 → 攒成卡片导出给 Anki**。
> 纪律：v0 只服务你自己、只支持 HN、复习交给现成 Anki，不自己造复习系统。

---

## 总架构

```
[content script]  监听划词、拿选区文本+位置、注入悬浮窗、朗读、攒卡
       │ 选中文本
       ▼
[service worker]  调 LLM(结构化JSON) —— 跨域 + 藏 key 都在这层
       │ 分析结果
       ▼
[content script]  悬浮窗渲染(词/语法/句式分区 + 🔊 + "存卡")
       │ 点"存卡"
       ▼
[chrome.storage] 累积卡片
       │ 点"导出"
       ▼
[下载 .txt]  → 你手动导入 Anki（一次性）
```

为什么 LLM 调用放 service worker：①绕开网页跨域；②**别让 API key 进 content script**（网页能读到就等于公开）。

## 分阶段实施（每阶段都能跑、能验证）

### Phase 0 · 骨架：划词 → 弹窗回显（先不接 LLM）
- MV3 `manifest.json`：`content_scripts` 注入 HN、`background` service worker、必要 `permissions`
- content script 监听 `mouseup` → `window.getSelection()` 拿文本，`getRangeAt(0).getBoundingClientRect()` 拿坐标
- 在选区附近注入一个悬浮 `<div>`，**先只显示选中的原文**
- ✅ 验证点：HN 上选中任意文字，旁边能冒出定位准确的悬浮窗。**最难的 UI 管道先打通**

> 关键：悬浮窗用 **Shadow DOM** 包起来，隔离样式，否则和 HN 的 CSS 互相污染。

### Phase 1 · 接 LLM 结构化分析
- content script 把选中文本发给 service worker（`chrome.runtime.sendMessage`）
- content script 先判断**选区是单词还是句子**（见下方"两种模式"），把模式一起发给 service worker
- service worker 调 LLM API，**prompt 强制返回 JSON**，且**面向零基础初学者**（见下方"分析原则"）。句子模式示例：
  ```json
  {
    "mode": "sentence",
    "translation_cn": "这次迁移缓解了大部分延迟尖峰，但在高负载下我们仍要应对缓存读到过期数据的问题。",
    "literal_cn": "这·迁移·缓解了·大部分·延迟尖峰，但·我们·仍然·不得不·应对·过期的·缓存读取·在高负载下",
    "structure_cn": "前半句讲好事，but（但是）一转，后半句讲还没解决的麻烦",
    "grammar_cn": ["but = 但是，连接两个相反的意思", "had to = 不得不（过去）"],
    "words": [
      {"word":"mitigate","pos":"动词","ipa":"/ˈmɪtɪɡeɪt/","meaning_cn":"减轻、缓解（让坏情况没那么糟）","example_en":"Caching mitigates database load.","example_cn":"缓存能减轻数据库压力。"}
    ]
  }
  ```
- 单词模式只回 `words`（且只一个词）；句子模式才回 `translation_cn / literal_cn / structure_cn / grammar_cn`
- 悬浮窗按模式渲染不同分区
- API key 阶段性做法：**v0 本地填**（自己用没问题）
- ✅ 验证点：选一个词、选一句话，分别出现对应详略的分析

### Phase 2 · 语音朗读
- 用浏览器自带 **Web Speech API**：`speechSynthesis.speak(new SpeechSynthesisUtterance(text))`
- 悬浮窗加 🔊 按钮，读选中句 / 读单个词
- 离线、免费、零成本；音质一般但 v0 够用（高音质云 TTS 留 v1）
- ✅ 验证点：点🔊能听到朗读

### Phase 3 · 攒卡 + 导出 Anki（你新增的需求，核心见下一节）
- 悬浮窗加"存为卡片"按钮 → 把卡片对象 push 进 `chrome.storage.local`
- 扩展弹窗(popup)里放"导出"按钮 → 生成 `.txt` → 触发下载
- ✅ 验证点：攒几张 → 导出 → 在 Anki 里一键导入成功

### Phase 4 · 打磨（可选，验证 v0 真用得上之后再做）
- 缓存（同词不重复问 LLM）+ 防抖，控制成本
- AnkiConnect 直推（见下）、云端 TTS、`.apkg` 带音频、支持更多网站

## 选区是单词还是句子（两种模式）

content script 先判断选区类型，再决定问 LLM 什么、悬浮窗显示什么、能生成几张卡。

**怎么判断**：用简单规则即可——选区是 1 个词或很短的短语（无句末标点、词数 ≤ 3）→ 单词模式；明显是一句话（多词 + 含句子结构）→ 句子模式。再给个手动切换按钮兜底。

| | 单词模式 | 句子模式 |
|---|---|---|
| 悬浮窗内容 | 紧凑：音标 + 词性 + 中文义 + 双语例句 | 丰富：整句翻译 + 逐词直译 + 句式大白话 + 语法点 + 重点词列表 |
| 存卡 | 1 张 vocab 卡 | 「存整句卡」→ 1 张 sentence 卡；重点词各自「存」→ N 张 vocab 卡 |
| 用途 | 读到生词顺手查 | 读到难句，既练句式、又把里面生词一起收 |

这样"一次划词产出多张卡"，比逐词查效率高得多，也对应 Anki 里 vocab / sentence 两种卡型（用 tag 区分）。

## 分析结果面向零基础初学者（核心原则）

> 目标：**一个对英语一窍不通的人，也能看懂每一条分析。** 这套写法对你自己也更省脑子，还顺带让工具将来能给真正的初学者用。

落地全在 LLM 的 prompt 里，把下面这些写成硬性要求：

1. **一切用中文解释**，不假设用户懂任何语法
2. **不用语法术语**；非用不可时立刻用大白话翻译——别说"介词短语"，说"一小段表示'在…情况下'的词"
3. **词性用中文**（动词 / 名词 / 形容词），不用 `vt.` `n.` 这种缩写
4. **每个词必带**：音标 + 一句简单英文例句 + **例句的中文翻译**（英文例句绝不裸给）
5. **句子给两版翻译**：①逐词直译（中英词对上，帮零基础建立对应）②自然翻译（通顺的中文）
6. **句式拆解讲"这句话在说什么、哪里转折"**，不是"主谓宾定状补"。例：「前半句讲好事，but（但是）一转，后半句讲还没解决的麻烦」
7. **例句贴近技术语境但用词简单**，别为了地道堆难词
8. **宁可少而懂，不要多而晕**：一次别塞超过 2–3 个语法点

prompt 里可以直接放一句定调：

```
你在帮一个完全不懂英语的中国初学者理解一段英文。
所有解释用中文；不要用语法术语，必须用时用大白话解释；
词性写"动词/名词"等中文；每个英文例句都要配中文翻译。
严格按给定 JSON 结构返回，不要多余文字。
```

> 顺带：存进 Anki 的卡片，背面也要带上**音标 + 词性 + 双语例句**，这样你以后复习时同样是零基础友好的，而不是只有一个干巴巴的中文词义。

## Anki 导出方案（重点）

**别一上来就做 `.apkg`。** 那是 Anki 的二进制包(内含 SQLite)，要拼媒体和数据库结构，复杂。最省事的 MVP 路线是：

### 首选：生成纯文本 `.txt`，Anki 原生导入

Anki 支持带**头部指令**的文本文件，一键设定牌组、模板、标签列。生成长这样（字段用 **Tab** 分隔）：

```
#separator:Tab
#html:true
#deck:HN English
#notetype:Basic
#tags column:3
mitigate<br><i>This strategy mitigates stale reads.</i>	减轻、缓解<br>例：The cache layer mitigates load spikes.	HN-vocab 2026-06-08
```

- 用 **Basic 模板**（只有 Front/Back 两字段），把"原句、例句"用 `<br>` 塞进去，配 `#html:true` 渲染
- `#tags column:3` 把第 3 列当标签（来源 + 日期），方便你之后按标签筛
- 单词卡、句式卡可以分别生成两份 txt，或用 tag 区分

> 这条路你只是在**拼字符串 + 触发下载**，几十行代码搞定。

### 致命坑：字段里的 Tab 和换行

TSV 最容易被内容里的制表符/换行打断。导出前必须清洗：
- 字段内的 `\t` → 替换成空格
- 字段内的换行 → 转成 `<br>`（已开 `#html:true`）
- 含特殊字符的字段用双引号包裹

### 进阶选项（v1 再考虑，二选一）

- **AnkiConnect 直推**：装个 AnkiConnect 插件后，Anki 桌面端会在 `localhost:8765` 暴露 HTTP API，扩展直接 POST `addNote` 把卡推进去，**免手动导入**。代价：要求 Anki 开着 + 装插件
- **genanki-js**：在浏览器里直接生成 `.apkg`，可打包音频（把 TTS 音频塞进卡）。代价：复杂度上一个台阶

你"待后续导入"的说法，正好匹配**首选的 txt 导出**——攒一批、导一次，不依赖 Anki 在线。

## 数据模型（攒卡用）

存进 `chrome.storage.local` 的卡片对象，建议：

```js
{
  id: "uuid",
  type: "vocab" | "sentence",   // 单词卡 / 句式卡
  front: "mitigate",
  back: "减轻、缓解",
  context: "This strategy mitigates stale reads.",  // 原句
  example: "The cache layer mitigates load spikes.",
  source: "Hacker News",
  url: "https://news.ycombinator.com/...",
  createdAt: "2026-06-08"
}
```

导出时把它拼成上面的 Anki txt 格式即可。type 决定进哪份/打哪个 tag。

## 几个非做不可的注意点

- **Manifest V3**：现在 Chrome 强制 MV3，background 是会休眠的 service worker，**不能在里面存长期状态**（状态放 storage）；照新文档写，别抄 MV2 老教程
- **API key 安全**：自己用本地填 OK；一旦想给别人用，必须搭个轻量后端中转，key 绝不进扩展包
- **成本控制**：每次划词都打 LLM，频繁用会累积费用 → 缓存 + 防抖
- **选区容错**：用户常选半个词、跨标签选，做好兜底

## v0 边界（再确认一遍）

✅ 做：HN 划词 → LLM 分析(词/语法/句式) → 悬浮窗 → Web Speech 朗读 → 攒卡 → 导出 txt 给 Anki
⛔ 先不做：支持所有网站、云端高音质 TTS、AnkiConnect/`.apkg`、账号同步、给别人用的后端、移动端、自建复习系统（复习永远交给现成 Anki）

## 和你整条线的关系

这个扩展**直接取代**之前那个独立 app 思路——因为它嵌在你真实读 HN 的场景里，摩擦最低、你会真的用。你只需做好"划词→分析→朗读→存卡→导出"这一段，**复习这件事 100% 外包给 Anki**，范围小一大圈。

> 老规矩：给 v0 设个硬时间盒（比如 2–3 周业余时间），按 Phase 0→3 走，跑通就先自己用起来。第一周自己没用到的功能，删掉。这东西最大的风险不是技术，是它膨胀成第六件分散精力的事。
